# 웹 애플리케이션 단계 구현 계획

## 1. 배경 및 목표
- `docs/SPEC.md`에 정의된 대로 Slack 중심 전달 단계를 웹 포털로 전환하여, 사용자가 관심 종목 구독 관리·리포트 열람·에이전트 대화를 한곳에서 처리할 수 있도록 한다.
- 수집/분석 파이프라인이 생산한 `ProcessedInsight`를 10분 이내에 웹에 게시하고, 리포트 열람 후 대화 전환율 30% 달성을 지원하는 사용자 경험을 구축한다.

## 2. 범위 및 비범위
- **범위**
  - 웹 프론트엔드(Next.js, TypeScript) 화면: 구독 관리, 리포트 목록/상세, 에이전트 채팅 패널
  - 웹 백엔드(API): 인증/인가, 리포트 조회, 종목 구독 CRUD, 채팅 세션/메시지 브로커
  - 게시(publish) 단계와의 연동: 뷰 모델/캐시 생성, SLA 모니터링 데이터 수집
  - 채팅 에이전트와의 인터페이스(OpenAI Assistants 혹은 Responses API)
- **비범위**
  - 데이터 수집·분석 파이프라인 로직 변경
  - 추가 알림 채널(이메일, 푸시 등) 구현
  - 고급 관리자 콘솔/대시보드 고도화

## 3. 주요 요구 요약
- 사용자는 종목을 검색해 구독/해지하고, 알림 시간대(`alert_window`)를 설정할 수 있어야 한다.
- 리포트 목록은 감성 태그, 날짜 필터, 즐겨찾기 기능을 제공하고, 상세 페이지는 요약, 감성 ASCII 게이지, 원문 링크, 아티팩트 뷰어를 포함한다.
- 대화형 에이전트 패널은 현재 리포트 컨텍스트를 바탕으로 WebSocket/SSE 스트리밍 응답을 제공해야 한다.
- 관리자 권한 사용자는 실패 리포트 재게시, Dead-letter 큐 확인 등 기본 운영 기능을 사용할 수 있어야 한다(SHOULD).

## 4. 기술 및 아키텍처 개요
- **프론트엔드**: Next.js(App Router) + TypeScript, 상태 관리는 Zustand/Redux Toolkit 후보, Tailwind 또는 Chakra UI 기반 디자인 시스템, WebSocket/SSE 클라이언트 포함.
- **백엔드 API**: FastAPI + SQLAlchemy, PostgreSQL + Redis 연동, OAuth2/SSO 기반 인증, OpenTelemetry 추적 수집.
- **채팅 에이전트**: OpenAI Assistants/Responses API 연동, Redis에 세션 컨텍스트 캐시, 장기 메시지는 `ChatMessage` 테이블에 적재.
- **배포**: 웹은 Vercel/Static Hosting(SSR 필요 시 Next.js 서버), API는 Docker 이미지로 ECS/Fargate 또는 Cloud Run, CI/CD는 GitHub Actions.
- **관찰성**: Prometheus/CloudWatch 메트릭(페이지 로드, API 지연, 채팅 전환율), 구조화 로그(JSON), SLA 위반은 PagerDuty/Slack Ops 알림.

## 6. 리스크 및 완화 전략
- **LLM 응답 품질 저하**: 사전 정의된 평가 프롬프트와 샘플 테스트를 자동화하고, 응답에 금칙어/PII 필터 적용.
- **실시간 채팅 지연**: 스트리밍 응답 적용, Redis 캐시 튜닝, 백프레셔 전략 마련.
- **대용량 리포트 렌더링 성능 문제**: Lazy loading과 캐시 계층(Edge Cache) 도입, 이미지 최적화.
